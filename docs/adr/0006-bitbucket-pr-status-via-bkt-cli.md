# Bitbucket Cloud pull-request status comes from the `bkt` CLI, not the REST API

Impala surfaces a **Worktree**'s **Pull request status** for GitHub by shelling out to the `gh` CLI, which owns its own credentials. To extend the same **Pull request status** to **Projects** whose **Remote provider** is Bitbucket Cloud, we source the data from the third-party `bkt` CLI (`avivsinai/bitbucket-cli`) rather than calling Bitbucket's REST API directly. A `RemoteProvider` detector classifies the `origin` URL (`github.com` → GitHub, `bitbucket.org` → Bitbucket Cloud, else unsupported) and dispatches to the matching populator; both fill the existing provider-neutral `PrInfo`, so the frontend, the TS types, and the PR-status commands are unchanged.

We chose `bkt` because it preserves the property that made the `gh` integration clean: **Impala stores no credential.** `bkt` keeps its token in the OS keychain and owns the token lifecycle (OAuth refresh, or year-long API tokens), so Impala never persists, refreshes, or sees a secret. Calling the REST API directly would force Impala to store a Bitbucket API token — and the existing precedent (`linearApiKey`) keeps it as plaintext in the SQLite `settings` table, a security downgrade from the keychain.

## Considered Options

- **`bkt` CLI (chosen):** keychain credentials, no stored secret, lifecycle owned by `bkt`, and `bitbucket.rs` mirrors `github.rs` (shell out, parse JSON, cache). Costs: a pre-1.0 single-maintainer dependency; `bkt pr list` has no `--head` filter (we list and match `source.branch.name` client-side) and a single-valued `--state` (query OPEN, fall back to MERGED); CI checks and line counts are separate `bkt` invocations.
- **Token + `reqwest` (REST API):** ~1–2 fewer calls per refresh — `pullrequests?q=source.branch.name="X"&state=…&fields=…` filters by branch and multiple states server-side and can expand `participants` (review decision) inline. But Impala would store the credential, own token expiry/401 handling, and maintain a REST client (pagination, field projections). Rejected: the call-count saving is modest and better addressed by poll-interval tuning, and it sacrifices the keychain / no-stored-secret guarantee.

## Consequences

- **Scope is Bitbucket Cloud only.** `bkt` also supports Bitbucket Data Center, but DC remotes live on arbitrary self-hosted domains the `origin`-URL detector can't classify; supporting them would require per-**Project** host configuration and is deliberately out of scope.
- **Fidelity is intentionally reduced (Tier B).** The badge shows existence / state / draft / branch / title / link plus the CI checks rollup (one extra `bkt pr checks` call). Review decision and additions/deletions are dropped — Bitbucket's `participants` approval model maps poorly onto GitHub's `reviewDecision`, both fields cost additional calls, and both are routinely empty in practice. `PrInfo.review_decision` and the diff counts already render conditionally, so they simply don't appear for Bitbucket PRs.
- **Polling is throttled for Bitbucket.** Bitbucket Cloud's per-user rate limit is tighter than the 60s all-**Worktrees** timer assumes at two calls per Worktree, so the Bitbucket path leans on focus and fs-change refreshes with a widened timer.
- **Users should authenticate `bkt` with an API token.** OAuth access tokens are short-lived (~2h); for a background-polling app the `BitbucketPane` steers users to `bkt auth login bitbucket.org --kind cloud --web-token` (valid up to a year) and surfaces the connected/expired state from `bkt auth status --json` so staleness is visible.
- **A new tool to install.** Like `gh`, `bkt` must be on PATH (resolved via `augmented_path()`); when absent the `BitbucketPane` shows install guidance. The `github_pr_status` cache table is renamed to the provider-neutral `pr_status` (pure cache — it repopulates, no data migration).
