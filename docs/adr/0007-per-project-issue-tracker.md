# A Project's issue tracker is a per-Project choice backed by a direct provider API

Impala lets each **Project** choose its **Issue tracker** — Linear, Jira, or none — rather than assuming Linear globally. Unlike **Remote provider**, which is auto-detected from the `origin` URL, the Issue tracker cannot be inferred from where the code is hosted (Linear and Jira are independent of the git remote), so it is an explicit per-Project setting stored alongside `baseBranch`. A backend `IssueTracker` trait resolves the project's tracker and credentials and dispatches to the matching implementation; the existing Linear path is refactored to flow through it, and a new Jira implementation talks directly to the Jira Cloud REST API (`/rest/api/3`). Issue search, the **Issue**→**Worktree** flow, the "move to In Progress" transition, the `docs/issues/<KEY>.md` context file, and the sidebar link are all provider-neutral.

## Considered Options

- **Direct Jira Cloud REST API, per-Project credentials (chosen):** mirrors how the other **Issue tracker** (Linear) already works — a stored credential and direct HTTPS calls. Jira's `site URL + account email + API token` is stored per-Project (scoped like `baseBranch`), because an agency works against a *different Jira site per client* and a single connection can't represent that. The frontend stops passing credentials; backend commands take a project/worktree path and resolve tracker + creds internally.
- **Jira via the `acli` CLI (rejected):** the obvious parallel to ADR-0006's `bkt`, keeping the credential in the OS keychain with no stored secret. Rejected because `acli` authenticates **one** global Atlassian identity, which structurally cannot model per-Project, per-client Jira sites — the exact requirement driving this feature. It would also add a runtime binary dependency and tie issue access to `acli`'s output format, where issue-tracker access in this app is otherwise the "direct API" lane.

## Consequences

- **Reconciliation with ADR-0006.** ADR-0006 deliberately avoided storing an Atlassian credential for **Pull request status**, choosing `bkt` for its keychain-owned token, and flagged the `linearApiKey` plaintext-SQLite precedent as a security downgrade. This ADR knowingly stores a Jira token in the SQLite `settings` table anyway. The decisions don't conflict: PR status can use a single global keychain login because there is one Bitbucket account; an **Issue tracker** must support per-client Jira sites, which no single global CLI login can express. The plaintext-token posture is accepted as consistent with the existing Linear precedent; OS-keychain storage is a possible future hardening.
- **Credential scope is asymmetric across the UI.** Linear's key stays global (Integrations settings); Jira's connection is configured per-Project (Project settings). The Project settings page shows a read-only pointer to global Integrations when Linear is the selected tracker.
- **Jira fidelity gaps vs Linear.** Jira returns descriptions and comments as ADF (a JSON node tree), not markdown, so the context writer converts a pragmatic ADF subset to markdown and degrades exotic nodes (tables, panels, media) to text. Inline attachment fetching — Linear-only today via `fetch_linear_attachment` — is **not** ported to Jira at launch. Jira provides no server-side branch name, so it is derived as `<KEY>-<slugified-title>` (editable); the issue **key** is kept uppercase so Bitbucket↔Jira branch linking recognizes it.
- **"In Progress" survives custom workflows.** Linear has a fixed state taxonomy; Jira workflows are per-project. The transition is selected by target **status category** (`indeterminate`), preferring an "In Progress"-named target, and is skipped silently when the current status offers no such transition — preserving the existing best-effort, fire-and-forget behavior.
- **Persisted link gains provider + url.** The provider-neutral `worktree_issues` row adds `provider` (so the post-creation context refresh dispatches to the right backend) and `url` (so the sidebar link no longer hardcodes `linear.app`). Existing rows backfill to `provider = 'linear'`, `url = https://linear.app/issue/<identifier>`.
- **No selection migration.** A Project's tracker resolves at read time: the stored per-Project setting if present, else Linear when a global Linear key is set, else none. Existing Linear users see no change; Jira-first users with no Linear key default to none and opt in per Project.
