Architecture Approaches

How should the Tauri app be structured internally?

A

Monolith Frontend

Single React app handles everything — sidebar, commit list, diff view, annotations. Tauri backend only provides git operations via Rust commands.

Pros

  • Simplest architecture
  • Fast to build v1
  • All state in one place

Cons

  • Can get complex as features grow
  • Git operations in Rust require more upfront work
B

Shell + Webview Panels

Tauri provides a native-feeling shell (sidebar, toolbar). Each diff opens in its own webview panel, like VS Code's editor tabs. Git operations via Rust commands.

Pros

  • Feels more native (window chrome from Tauri)
  • Each diff is isolated — no cross-contamination
  • Can open multiple diffs side by side

Cons

  • Multi-webview is more complex in Tauri
  • Harder to share state between panels
C

React App + Git CLI Bridge (Recommended)

Single React app for the UI. Tauri backend is a thin bridge that shells out to git CLI for all git operations (worktree list, log, diff). Avoids the complexity of a Rust git library.

Pros

  • Fast to build — no Rust git library needed
  • Git CLI is battle-tested and complete
  • React handles all UI complexity
  • Easy to add new git operations

Cons

  • Requires git installed on the system
  • CLI parsing can be fragile (mitigated with --porcelain flags)
